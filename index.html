Cómo se mide DevOps: las 4 métricas DORA

🚀 Frecuencia de despliegue

Cada cuánto liberas a producción. Más frecuente = más ágil (tú: despliegues diarios).

⏱️ Lead time for changes

Cuánto tarda un cambio desde el commit hasta producción.

🔧 MTTR — tiempo de recuperación

Cuánto tardas en restablecer el servicio tras un fallo.

❌ Change failure rate

% de despliegues que causan un fallo. Menos = más calidad.

SRE en una frase: defines SLIs/SLOs (objetivos de fiabilidad), gastas un error budget cuando fallas, y haces postmortems sin culpa para aprender, no para señalar.
Cómo lo cuentas tú: "Trabajo DevOps de punta a punta: automatización con CI/CD, contenedores en Kubernetes, infraestructura como código y observabilidad, con cultura de mejora continua. Mis resultados hablan en métricas DORA: despliegues diarios, −60% en tiempo de despliegue y zero-downtime."
¿Qué es DevOps, en una frase?

Unir desarrollo y operaciones con cultura, automatización y medición para entregar software más rápido y confiable. No es un cargo ni una herramienta: es una forma de trabajar (modelo CALMS).

¿Qué es Infraestructura como Código (IaC)?

Definir servidores, redes y recursos en archivos versionados (Terraform, Bicep) en lugar de configurarlos a mano. Así la infraestructura es reproducible, revisable y auditable, y se despliega igual en cada entorno.

¿Qué miden las métricas DORA?

La salud de la entrega: frecuencia de despliegue, lead time de un cambio, MTTR (recuperación ante fallos) y change failure rate. Son el estándar para medir a un equipo DevOps de alto rendimiento.

El código Microservicios con Java & Spring Boot

Antes de empaquetar y desplegar, hay que escribir el servicio. Spring Boot es el framework más usado para construir microservicios en Java: trae un servidor embebido (Tomcat) y configuración por defecto, así levantas una API REST en minutos y te concentras en la lógica de negocio.

Analogía: Spring Boot te entrega el kit de arranque ya ensamblado —servidor web, inyección de dependencias, acceso a datos— para que no reinventes la plomería y escribas solo lo que aporta valor.

Las anotaciones que debes saber explicar

@SpringBootApplication

Marca el punto de arranque; activa autoconfiguración y el escaneo de componentes.

@RestController

Expone la clase como API REST: sus métodos devuelven JSON directamente.

@Service

Contiene la lógica de negocio; es un componente inyectable.

@Repository

Acceso a datos. Con Spring Data JPA casi no escribes SQL.

@Entity

Mapea una clase Java a una tabla (ORM con Hibernate).

Inyección por constructor

Spring "inyecta" las dependencias que la clase necesita: desacopla y facilita las pruebas.

Un microservicio completo, capa por capa

Ejemplo: un servicio de facturas con un endpoint para listar pendientes y otro para crear.

1. Entity — el objeto que mapea a la tabla:

@Entity
public class Factura {
  @Id @GeneratedValue
  private Long id;
  private String concepto;
  private BigDecimal monto;
  private LocalDate vence;
  // + getters / setters
}

2. Repository — acceso a datos sin escribir SQL (Spring lo genera desde el nombre del método):

public interface FacturaRepository
        extends JpaRepository<Factura, Long> {

  List<Factura> findByVenceBefore(LocalDate fecha);  // SELECT ... WHERE vence < ?
}

3. Service — la lógica de negocio, con inyección por constructor:

@Service
public class FacturaService {
  private final FacturaRepository repo;

  public FacturaService(FacturaRepository repo) {  // Spring inyecta el repo
    this.repo = repo;
  }

  public List<Factura> pendientes() {
    return repo.findByVenceBefore(LocalDate.now());
  }

  public Factura crear(Factura f) { return repo.save(f); }
}

4. Controller — expone los endpoints REST:

@RestController
@RequestMapping("/api/facturas")
public class FacturaController {
  private final FacturaService service;

  public FacturaController(FacturaService service) {
    this.service = service;
  }

  @GetMapping("/pendientes")
  public List<Factura> pendientes() {
    return service.pendientes();
  }

  @PostMapping
  public ResponseEntity<Factura> crear(@Valid @RequestBody Factura f) {
    return ResponseEntity.status(201).body(service.crear(f));
  }
}

5. application.yml — configuración y conexión a la base de datos (sin credenciales en el código):

spring:
  datasource:
    url: jdbc:postgresql://db:5432/rohu
    username: ${DB_USER}      # viene de variables/Secrets
    password: ${DB_PASS}

Preguntas típicas de Spring Boot

¿Qué es la inyección de dependencias y cómo la hace Spring?

Es dejar que un contenedor (el IoC container de Spring) cree y "entregue" las dependencias que una clase necesita, en vez de que ella las construya. Spring detecta los componentes (@Service, @Repository) y los inyecta — preferiblemente por constructor, lo que desacopla y hace el código fácil de probar (puedes inyectar un mock).

¿Cómo accedes a la base de datos sin escribir SQL?

Con Spring Data JPA: extiendes JpaRepository<Entidad, Id> y ya tienes save, findAll, findById, etc. Además, Spring genera consultas a partir del nombre del método (findByVenceBefore → WHERE vence < ?). Para casos complejos usas @Query con JPQL o SQL nativo.

¿Cómo conecta un servicio Spring Boot con los health checks de Kubernetes?

Con Spring Boot Actuator, que expone /actuator/health/readiness y /actuator/health/liveness. Las probes del Deployment de Kubernetes apuntan a esas URLs: si readiness no responde OK, K8s no le manda tráfico; si liveness falla, lo reinicia. Eso es lo que habilita el zero-downtime real.

¿Cómo se comunican dos microservicios en Spring Boot?

De forma síncrona con WebClient u OpenFeign (llamadas REST), envolviéndolas en un circuit breaker (Resilience4j) para tolerar fallos; o de forma asíncrona por eventos/mensajería (NATS), que los desacopla y los hace más resilientes.

El frontend Cursillo de Next.js

Next.js es el framework de React para producción: enrutado, renderizado en servidor y hasta APIs — todo listo. Es la capa de UI con la que tus microservicios se vuelven una app real (como tu ROHU PayControl PWA).

Analogía: si React son los ladrillos (componentes de UI), Next.js es la casa prefabricada: te da enrutado, renderizado, optimización y estructura de producción, para no ensamblar todo a mano.

Los conceptos que debes saber explicar

App Router

Enrutado por carpetas: cada carpeta en app/ es una ruta; page.tsx es la página y layout.tsx el marco compartido.

Server vs Client

Por defecto los componentes corren en el servidor (rápidos, sin JS al cliente). "use client" los vuelve interactivos en el navegador.

Estrategias de render

SSR (en cada request), SSG (en el build), ISR (regenera cada X), CSR (en el cliente). Eliges por página.

Data fetching

En Server Components haces await fetch(...) directo — sin useEffect. Con caché y revalidación integradas.

Route Handlers (API)

app/api/.../route.ts crea endpoints (tu propio backend ligero) dentro de Next.js.

Carga y errores

loading.tsx y error.tsx dan spinners y manejo de errores automáticos por ruta.

Preguntas típicas de Next.js

¿Por qué Next.js y no solo React?

React es una librería de UI; Next.js le agrega lo de producción: enrutado por archivos, renderizado en servidor (mejor rendimiento y SEO), optimización de imágenes/fuentes, code-splitting y API routes — sin configurarlo todo a mano.

¿Server Components vs Client Components?

Los Server Components (por defecto) se renderizan en el servidor: no envían JS al navegador y pueden hacer fetch/leer datos directo — ideales para contenido. Los Client Components ("use client") corren en el navegador y permiten interactividad (estado, eventos, hooks como useState).

¿SSR, SSG o ISR — cuándo usar cada uno?

SSG (genera en el build) para contenido estático; SSR (en cada request) para datos siempre frescos o personalizados; ISR (regenera cada cierto tiempo) para el punto medio: rápido como estático pero se actualiza. Se elige por página con las opciones de fetch/revalidate.

Backend JS Cursillo de Node.js

JavaScript del lado del servidor. Es el motor detrás de Next.js y una opción ágil para APIs y servicios ligeros — complemento a tu backend Java/Spring y Python.

La clave de Node: un solo hilo con un event loop y E/S no bloqueante. En vez de esperar parado a que responda la base de datos o la red, sigue atendiendo otras peticiones y reacciona cuando el dato llega. Por eso brilla en apps con mucha E/S y muchas conexiones.

Conceptos que debes explicar

Event loop

El bucle que atiende eventos y callbacks sin bloquear. Un hilo, alta concurrencia de E/S.

No bloqueante / async

Las operaciones de E/S devuelven promesas; con async/await escribes asíncrono que se lee como síncrono.

npm

El gestor de paquetes (el ecosistema más grande). package.json declara dependencias y scripts.

Express / Nest

Express: framework web minimalista. NestJS: estructurado (al estilo Spring) para apps grandes.

Módulos

import/export (ESM) para organizar el código; ejecución con Node o runtimes como Bun/Deno.

¿Cuándo Node?

APIs de mucha E/S, tiempo real (websockets), BFF para el frontend. Para CPU intensivo, mejor otro stack.

Una API REST mínima (Express)

import express from "express";
const app = express();
app.use(express.json());

// GET no bloqueante: espera la BD sin frenar el servidor
app.get("/facturas/pendientes", async (req, res) => {
  const data = await repo.pendientes();
  res.json(data);
});

app.listen(3000, () => console.log("API en :3000"));

Node vs. Java/Spring

Node: arranque rápido, ideal para E/S y tiempo real, mismo lenguaje que el frontend. Spring: tipado fuerte, maduro para dominios complejos y transaccionales. Se complementan.

Node y Next.js

Next.js corre sobre Node; sus Route Handlers y el renderizado en servidor son, por debajo, Node ejecutando tu JavaScript.

Cómo lo cuentas tú: "Uso Node.js para APIs ligeras y BFF que alimentan el frontend Next.js, aprovechando su modelo no bloqueante para operaciones de mucha E/S — complementando mis microservicios Java/Spring y Python según lo que cada caso necesita."

Preguntas típicas de Node.js

¿Por qué Node es de un solo hilo y aun así maneja mucha concurrencia?

Porque no bloquea: mientras espera una operación de E/S (BD, red, archivo), el event loop sigue atendiendo otras peticiones y ejecuta el callback cuando el resultado llega. Así, un solo hilo sostiene miles de conexiones concurrentes de E/S.

¿Diferencia entre bloqueante y no bloqueante?

Una operación bloqueante detiene el hilo hasta terminar; una no bloqueante devuelve el control de inmediato y avisa después (callback/promesa). Node es no bloqueante por diseño, por eso conviene evitar código CPU-intensivo que "tape" el event loop.

¿Cuándo elegirías Node y cuándo Java/Spring?

Node para APIs de mucha E/S, tiempo real (websockets) o un BFF en el mismo lenguaje del frontend. Java/Spring para dominios complejos, tipado fuerte y transaccionalidad pesada. No es "uno u otro": elijo por el problema.

Los datos Cursillo de Bases de Datos & SQL

El corazón de casi toda app: guardar y consultar datos de forma correcta y rápida. Tú trabajas PostgreSQL y Oracle a diario (dashboards, reconciliación) — aquí lo estructuras para explicarlo.

Modelo relacional en una frase: los datos viven en tablas (filas y columnas); las tablas se relacionan por claves (una primary key identifica cada fila, una foreign key apunta a otra tabla).

Conceptos que debes dominar

Claves

Primary key: identifica cada fila. Foreign key: relaciona tablas y garantiza integridad referencial.

JOINs

Combinan tablas: INNER (solo coincidencias), LEFT (todo lo de la izquierda + lo que empate).

Índices

Aceleran búsquedas (como el índice de un libro). Clave para el rendimiento; cuestan en escritura.

Transacciones (ACID)

Atomicidad, Consistencia, aIslamiento, Durabilidad. O todo o nada — clave en finanzas.

Normalización

Organizar tablas para evitar datos duplicados e inconsistentes (1FN, 2FN, 3FN).

Problema N+1

Hacer 1 consulta + N por cada fila. Se evita con JOINs o carga anticipada (eager). Clásico con ORMs.

SQL esencial en un ejemplo

-- Facturas vencidas por cliente, con su nombre (JOIN + filtro + orden)
SELECT c.nombre, f.concepto, f.monto, f.vence
FROM   facturas f
JOIN   clientes c ON c.id = f.cliente_id
WHERE  f.pagada = false
  AND  f.vence < CURRENT_DATE
ORDER BY f.vence;

-- Agregación: total pendiente por cliente (solo > 1.000.000)
SELECT c.nombre, SUM(f.monto) AS total
FROM   facturas f JOIN clientes c ON c.id = f.cliente_id
WHERE  f.pagada = false
GROUP BY c.nombre
HAVING SUM(f.monto) > 1000000;

SQL vs. NoSQL

SQL (relacional): estructura fija y transacciones fuertes — ideal para datos financieros. NoSQL: flexible y escala horizontal — para grandes volúmenes o esquema variable.

Pool de conexiones

Reutilizar conexiones abiertas (HikariCP en Spring) en vez de abrir una por request — clave para el rendimiento bajo carga.

Cómo lo cuentas tú: "Modelo y consulto PostgreSQL y Oracle en apps de misión crítica: diseño esquemas normalizados, optimizo consultas con índices, manejo transacciones ACID para la reconciliación financiera y evito el problema N+1 con Spring Data JPA — procesando miles de registros diarios."

Preguntas típicas de bases de datos

¿Diferencia entre INNER JOIN y LEFT JOIN?

INNER devuelve solo las filas que coinciden en ambas tablas. LEFT devuelve todas las de la tabla izquierda y, donde no hay coincidencia, rellena con NULL — útil para listar clientes aunque no tengan facturas.

¿Para qué sirve un índice y cuándo NO usarlo?

Acelera lecturas al evitar escanear toda la tabla. Pero ralentiza las escrituras (hay que mantenerlo) y ocupa espacio; no conviene indexar columnas que casi no se consultan ni tablas muy pequeñas. Se indexan las columnas de filtros/JOINs frecuentes.

¿Qué significa ACID?

Las garantías de una transacción: Atomicidad (todo o nada), Consistencia (deja la BD en estado válido), Aislamiento (transacciones concurrentes no se pisan) y Durabilidad (una vez confirmada, persiste). Es lo que hace confiable una operación financiera.

Mensajería Cursillo de NATS (event-driven)

Cómo se comunican tus microservicios sin acoplarse. NATS es el sistema de mensajería que usaste en la plataforma de compras y recargas para que los servicios hablen por eventos.

Analogía: en vez de que un servicio llame por teléfono a otro (y espere), publica un anuncio (evento); quien le interese lo escucha. Emisor y receptor no necesitan conocerse ni estar activos a la vez — eso es estar desacoplados.

Los conceptos

Publish / Subscribe

Un servicio publica en un subject (tema) y todos los suscritos lo reciben. Uno-a-muchos.

Subject

El "canal" con nombre jerárquico (p. ej. pagos.recarga.ok). Admite comodines para suscribirse a grupos.

Queue groups

Varios consumidores comparten la carga: cada mensaje lo procesa solo uno del grupo (balanceo).

Request / Reply

NATS también soporta pedir y esperar respuesta (RPC ligero), no solo "disparar y olvidar".

JetStream

La capa de persistencia: guarda mensajes, reintenta y garantiza entrega (at-least-once). Para no perder eventos.

¿Por qué event-driven?

Desacopla, absorbe picos, agrega consumidores sin tocar al emisor y da resiliencia si algo se cae.

La idea en dos líneas

// El servicio de recargas PUBLICA un evento cuando algo pasa
nc.publish("pagos.recarga.ok", evento);

// Otro servicio (notificaciones) SE SUSCRIBE y reacciona
nc.subscribe("pagos.recarga.ok", msg -> enviarNotificacion(msg));

NATS vs. Kafka / RabbitMQ

NATS: ultraligero, rapidísimo y simple. Kafka: para grandes volúmenes y re-lectura de historiales. RabbitMQ: enrutamiento rico con colas.

Garantía de entrega

NATS "core" es at-most-once (rápido, sin persistir); con JetStream pasa a at-least-once con almacenamiento.

Dónde encaja: es el "sistema nervioso" entre tus microservicios en AKS. Complementa las REST APIs (síncronas) con eventos asíncronos para todo lo que no necesita respuesta inmediata.
Cómo lo cuentas tú: "Diseñé la comunicación event-driven de la plataforma con NATS: los microservicios publican eventos y otros los consumen de forma desacoplada (pub/sub y queue groups), lo que dio resiliencia, absorbió picos de tráfico y permitió agregar servicios sin tocar a los existentes."

Preguntas típicas de mensajería

¿Por qué usar mensajería en vez de llamadas REST directas?

Para desacoplar: el emisor no depende de que el receptor esté disponible ni de quién sea. Da resiliencia (si un servicio cae, el mensaje espera), absorbe picos y permite escalar consumidores. REST se queda para lo síncrono (necesito la respuesta ya); los eventos para lo asíncrono.

¿Diferencia entre pub/sub y queue group?

En pub/sub puro, todos los suscritos reciben cada mensaje (uno-a-muchos). En un queue group los consumidores comparten la carga y cada mensaje lo procesa solo uno — así escalas el procesamiento horizontalmente.

¿NATS garantiza que no se pierda un mensaje?

El NATS "core" es at-most-once (muy rápido, pero si nadie escucha, el mensaje se pierde). Para garantía de entrega usas JetStream, que persiste los mensajes y reintenta (at-least-once) — el que usarías para eventos críticos como un pago.

Paso 1 Docker & Dockerfile

Docker resuelve el clásico “en mi máquina sí funciona”: empaqueta tu aplicación con TODO lo que necesita para correr, de forma aislada y reproducible.

Analogía: un contenedor es como una lonchera sellada: adentro va tu comida (la app) y sus cubiertos (librerías, versión de Python/Java, config). La abras donde la abras —tu portátil o un servidor en Azure— adentro está exactamente lo mismo.

Los tres conceptos que debes distinguir

Dockerfile

la receta

Un archivo de texto con instrucciones para construir la imagen (qué base usar, qué copiar, qué instalar, cómo arrancar).

Imagen

la plantilla

El resultado de construir el Dockerfile. Es inmutable y se guarda en un registro (p. ej. Azure Container Registry).

Contenedor

la instancia viva

Una imagen en ejecución. De una misma imagen puedes levantar 1 o 100 contenedores iguales.

Cómo se construye por “capas”

Cada instrucción del Dockerfile crea una capa que Docker cachea. Si no cambia, se reutiliza → los builds son rápidos. Por eso se ponen primero las cosas que cambian poco (dependencias) y de último el código.

CMD ["uvicorn", ...]cómo arranca
COPY . .tu código (cambia seguido)
RUN pip install ...dependencias (cacheable)
COPY requirements.txtlista de dependencias
FROM python:3.12-slimimagen base (SO + runtime)

Dockerfile de ejemplo (un microservicio Python/FastAPI)

FROM python:3.12-slim            # base: Linux mínimo + Python
WORKDIR /app                     # carpeta de trabajo dentro del contenedor
COPY requirements.txt .          # primero las dependencias (capa cacheable)
RUN  pip install --no-cache-dir -r requirements.txt
COPY . .                         # luego el código de la app
EXPOSE 8000                       # puerto que expone el servicio
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Flujo básico

docker build (crea la imagen desde el Dockerfile) → docker push (la sube al registro) → docker run (levanta el contenedor). En producción, ese “run” lo hace Kubernetes por ti.

Cómo lo cuentas tú: “Empaqueté cada microservicio de la plataforma de compras y recargas en su propia imagen Docker, de modo que corriera idéntico en desarrollo y en producción, y lo publiqué en Azure Container Registry para desplegarlo en AKS.”

Paso 2 Kubernetes (K8s)

Un contenedor está bien para una app; pero con decenas de microservicios, en muchos servidores, escalando y fallando en vivo, coordinarlos a mano es imposible. Kubernetes orquesta todo eso automáticamente.

Analogía: Kubernetes es el director de orquesta. Tú le dices el resultado que quieres (“quiero 5 copias de este servicio, siempre sanas y accesibles”) y él se encarga de reiniciar lo que se cae, repartir la carga y actualizar sin que el público note el cambio.

El modelo mental clave: estado deseado

No le das órdenes paso a paso; le entregas un manifiesto YAML que declara el estado deseado. Kubernetes lo reconcilia constantemente: si algo se desvía (un Pod muere), lo vuelve a poner como pediste. Eso es la auto-reparación.

Los objetos que debes nombrar

Pod

la unidad mínima

Envuelve uno o varios contenedores que viven juntos. Es lo más pequeño que Kubernetes despliega.

Deployment

réplicas + actualizaciones

Declara cuántas copias (réplicas) quieres de un Pod y gestiona las actualizaciones graduales (rolling updates).

Service

dirección estable

Da una IP/DNS fija para llegar a un grupo de Pods y balancea el tráfico entre ellos, aunque los Pods cambien.

Ingress

puerta de entrada

Enruta el tráfico HTTP/HTTPS desde afuera hacia el Service correcto (por dominio o ruta).

ConfigMap & Secret

config fuera del código

Guardan configuración y credenciales por separado, para no “quemarlas” en la imagen.

HPA (autoscaler)

escala solo

Sube o baja el número de réplicas según la carga (CPU/memoria) — clave bajo alto tráfico.

Cómo se despliega

# Declaras el estado deseado y lo aplicas
kubectl apply -f deployment.yaml
kubectl get pods           # ver las réplicas corriendo
kubectl rollout status ... # seguir una actualización

Las tres superpotencias (tu argumento de venta)

♻️ Rolling update

Levanta las réplicas nuevas y retira las viejas de a poco → zero-downtime.

🩹 Auto-reparación

Si un Pod se cae o no responde (health checks), lo reinicia o reemplaza solo.

📈 Autoescalado

Con el HPA, agrega réplicas cuando sube el tráfico y las quita cuando baja.

Dato de entrevista — probes: el zero-downtime real depende de las readiness y liveness probes. La readiness le dice a Kubernetes “ya estoy listo para recibir tráfico” (no enruta hasta entonces); la liveness dice “sigo vivo” (si falla, me reinicia). Mencionarlas demuestra profundidad.
Cómo lo cuentas tú: “Desplegué los microservicios en AKS con Deployments y rolling updates, logrando despliegues sin downtime; el autoescalado (HPA) sostuvo tiempos de respuesta de 500 ms bajo alto tráfico, y la auto-reparación mantuvo la alta disponibilidad.”

A fondo Kubernetes por dentro

Si el entrevistador va más allá de "¿qué es un Pod?", esto es lo que separa a alguien que usa Kubernetes de alguien que lo entiende.

Arquitectura del clúster

Un clúster tiene dos mitades: el control plane (el cerebro que decide) y los nodos (los músculos que ejecutan tus contenedores).

🧠 Control plane

decide y coordina

API Server (la puerta de entrada), etcd (la base de datos del estado), Scheduler (asigna Pods a nodos) y Controller Manager (reconcilia el estado deseado).

💪 Nodos (workers)

ejecutan los Pods

kubelet (agente que crea los contenedores), kube-proxy (redes/enrutamiento) y el container runtime (containerd) que realmente corre los contenedores.

En AKS, Azure gestiona el control plane por ti (no lo ves ni lo pagas como VMs); tú solo administras los nodos y tus aplicaciones.

Más objetos que vale la pena nombrar

StatefulSet

Para apps con estado e identidad estable (bases de datos, colas). A diferencia del Deployment, cada Pod es distinguible y persistente.

DaemonSet

Garantiza un Pod por nodo — típico para agentes de logs, métricas o seguridad.

Job / CronJob

Tareas que corren y terminan; el CronJob las programa (p. ej. una conciliación nocturna).

PV / PVC

PersistentVolume y su reclamo (PVC): almacenamiento que sobrevive aunque el Pod muera.

Namespace

Divide el clúster en espacios lógicos (dev / qa / prod, o por equipo).

Service types

ClusterIP (interno), NodePort (puerto del nodo), LoadBalancer (IP pública, vía Azure LB).

Salud, recursos y actualización sin cortes

Requests & Limits

requests = lo que el Pod reserva; limits = su tope. Evitan que un servicio ahogue a los demás.

Probes

readiness (¿listo para tráfico?), liveness (¿sigo vivo?) y startup (para arranques lentos).

Estrategia rolling

maxUnavailable: 0 + maxSurge: 1 = nunca bajas capacidad mientras actualizas.

apiVersion: apps/v1
kind: Deployment
metadata: { name: recargas-svc }
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }   # zero-downtime
  template:
    spec:
      containers:
        - name: app
          image: acr.azurecr.io/recargas:1.4.2
          resources:
            requests: { cpu: "250m", memory: "256Mi" }
            limits:   { cpu: "500m", memory: "512Mi" }
          readinessProbe: { httpGet: { path: /health, port: 8000 } }
          livenessProbe:  { httpGet: { path: /alive,  port: 8000 } }

kubectl — comandos en la punta de la lengua

kubectl get pods -n prod            # listar Pods de un namespace
kubectl describe pod <nombre>       # diagnosticar por qué falla
kubectl logs -f <pod>               # ver logs en vivo
kubectl rollout status deploy/recargas-svc
kubectl rollout undo   deploy/recargas-svc   # revertir un despliegue
kubectl scale deploy/recargas-svc --replicas=8
Pregunta típica: ¿qué pasa cuando un nodo se cae?

El control plane lo detecta (deja de recibir el "latido" del kubelet), marca sus Pods como perdidos y el Scheduler los reprograma en nodos sanos. Si tenías varias réplicas repartidas, el servicio ni se entera — eso es alta disponibilidad real.

Empaquetado Helm

Mantener a mano decenas de archivos YAML —y duplicarlos por cada entorno— es un dolor. Helm es el gestor de paquetes de Kubernetes: el "apt" o "npm" del clúster.

Analogía: un Chart de Helm es una plantilla parametrizable de tu despliegue. En vez de un YAML para dev y otro para prod, tienes una plantilla y un archivo de values por entorno: cambias valores, no copias archivos.

Los tres términos

Chart

el paquete

Conjunto de plantillas YAML + valores por defecto. Reutilizable y versionable.

values.yaml

la configuración

Los valores que rellenan la plantilla (imagen, réplicas, dominio). Uno por entorno.

Release

la instalación

Una instancia de un chart desplegada, con historial de versiones para hacer rollback.

Estructura de un chart

mi-servicio/
├─ Chart.yaml         # nombre y versión del chart
├─ values.yaml        # valores por defecto
├─ values-prod.yaml   # valores de producción
└─ templates/
   ├─ deployment.yaml # usa {{ .Values.replicas }}, {{ .Values.image.tag }}
   ├─ service.yaml
   └─ ingress.yaml

Comandos clave

helm install recargas ./mi-servicio -f values-prod.yaml
helm upgrade recargas ./mi-servicio --set image.tag=1.4.3   # nueva versión
helm rollback recargas 2      # volver a una versión anterior en segundos
helm list                     # ver releases instaladas
Cómo lo cuentas tú: "Empaqueté los microservicios como charts de Helm para desplegarlos de forma consistente en dev, QA y producción cambiando solo los values, con versionado y rollback inmediato — despliegues repetibles y seguros."

Paso 3 Azure & AKS

Azure es la nube de Microsoft: servidores, redes, bases de datos y servicios bajo demanda. AKS (Azure Kubernetes Service) es Kubernetes gestionado: Azure opera la parte compleja (el “control plane”) y tú te concentras en desplegar tus apps.

Azure DevOps / Pipelines

secretos + observabilidad

Conceptos que debes manejar

La jerarquía del pipeline: etapas → trabajos → pasos concretos.

Paso 5 Calidad & seguridad del código: SonarQube y Kiuwan

En banca y gobierno, la calidad y la seguridad del código no son opcionales: se automatizan dentro del pipeline como un "quality gate" que bloquea el despliegue si el código no cumple.

Concepto base — SAST: ambas herramientas hacen análisis estático (Static Application Security Testing): revisan el código sin ejecutarlo, buscando bugs, vulnerabilidades y malas prácticas. Se contrasta con DAST, que prueba la aplicación ya corriendo.

SonarQube

Qué detecta

Bugs, vulnerabilidades, "code smells" (malas prácticas), duplicación y cobertura de pruebas. Cuantifica la deuda técnica.

Quality Gate

Un umbral que defines (p. ej. "0 vulnerabilidades nuevas, cobertura ≥ 80%"). Si no se cumple, el pipeline falla y no se despliega.

Kiuwan

Enfoque

Plataforma de seguridad y calidad orientada a compliance. Fuerte en Code Security (SAST) y SCA (análisis de dependencias con vulnerabilidades conocidas).

Estándares

Mapea hallazgos a OWASP Top 10, CWE, PCI-DSS, CERT… — clave en entornos regulados como banca y gobierno.

Pregunta típica: ¿dónde encajan estas herramientas en el pipeline?

Justo después de las pruebas y antes de construir/desplegar la imagen: el pipeline manda el código al análisis, espera el resultado del Quality Gate y solo continúa si aprueba. Así se evita que código con vulnerabilidades o baja calidad llegue a producción.

La calidad Cursillo de Pruebas (Testing)

Probar es lo que hace que el "quality gate" tenga sentido: código que se puede cambiar sin miedo. Complementa tu sección de SonarQube/Kiuwan con las pruebas que alimentan la cobertura.

La pirámide de pruebas: muchas unitarias (rápidas, aisladas) en la base, menos de integración (varias piezas juntas) en el medio, y pocas end-to-end (todo el sistema) arriba. Rápido y barato abajo; lento y costoso arriba.

Los tipos y conceptos

Unitarias

Prueban una clase/método aislado. Rápidas; usan mocks para las dependencias.

Integración

Prueban varias piezas juntas (p. ej. servicio + base de datos real).

End-to-end (E2E)

Prueban el flujo completo como un usuario (UI → API → BD).

Mock vs. Stub

Mock: verifica interacciones (¿se llamó?). Stub: devuelve datos fijos. Aíslan la unidad bajo prueba.

Cobertura

% de código ejecutado por las pruebas. Útil como señal, no como meta ciega.

TDD

Red → Green → Refactor: escribes la prueba que falla, el código mínimo para pasarla, y luego limpias.

Una prueba unitaria (JUnit + Mockito)

@Test
void pendientes_devuelveSoloVencidas() {
  // Arrange — el repositorio es un mock
  when(repo.findByVenceBefore(any()))
      .thenReturn(List.of(new Factura("Luz", 50000)));
  // Act
  var result = service.pendientes();
  // Assert
  assertEquals(1, result.size());
  verify(repo).findByVenceBefore(any());
}

@SpringBootTest

Levanta el contexto de Spring para pruebas de integración del servicio real y sus beans.

Testcontainers

Arranca una base de datos real en un contenedor Docker durante las pruebas — integración fiel, no un simulacro.

Conecta con tu pipeline: estas pruebas corren en el paso de Test del CI/CD; su cobertura alimenta el Quality Gate de SonarQube/Kiuwan que decide si el código pasa a producción.

Preguntas típicas de testing

¿Qué es la pirámide de pruebas?

Una guía de proporción: muchas unitarias (rápidas y baratas), algunas de integración, y pocas end-to-end (lentas y frágiles). Evita el "cono de helado" (demasiadas E2E), que es lento e inestable.

¿Diferencia entre mock y stub?

Un stub devuelve respuestas predefinidas para que la prueba avance; un mock además verifica que se le llamó como esperabas (interacciones). Ambos aíslan la unidad de sus dependencias.

¿Qué es TDD?

Test-Driven Development: escribes primero una prueba que falla (Red), luego el código mínimo para que pase (Green) y después refactorizas (Refactor). Fuerza un diseño testeable y evita sobre-ingeniería.

Curso Liderazgo y gestión de equipos

Cómo planear el trabajo, estimar, hacer seguimiento y desarrollar personas. Tips prácticos que ya aplicas liderando equipos de hasta 21 personas — para contarlos con estructura.

1 · Planear las asignaciones

Desglosa el trabajo (WBS)

Parte lo grande en tareas pequeñas y claras (1–3 días). Lo pequeño se estima mejor, se sigue mejor y motiva por avances visibles.

Prioriza

MoSCoW (Must/Should/Could/Won't) o matriz valor vs. esfuerzo: primero lo de más valor y menos esfuerzo.

Asigna por fortalezas y capacidad

Empareja la tarea con quien la hace mejor, respetando su carga real (no al 100%: deja holgura para imprevistos).

Mapea dependencias

Identifica qué bloquea a qué para secuenciar y evitar cuellos de botella.

2 · Estimar (sin adivinar)

T-shirt sizing

S/M/L/XL para una idea rápida de tamaño relativo antes de afinar.

Story points + Planning Poker

El equipo estima por complejidad relativa, en consenso — no en horas. Reduce el sesgo individual.

Estimación a 3 puntos (PERT)

(optimista + 4×probable + pesimista) / 6. Incorpora la incertidumbre.

Falacia de planificación

Tendemos a subestimar. Agrega buffer y compara con datos históricos.

Re-estima con velocity

Usa cuánto entregó el equipo en sprints pasados para proyectar lo que cabe en el próximo.

Estima en equipo

Quien hace el trabajo estima; el líder facilita, no impone.

3 · Seguimiento de tareas

Tablero Kanban

Visualiza el flujo (Por hacer → En curso → En revisión → Hecho). El estado se ve de un vistazo.

Límites WIP

Limita el trabajo en curso: menos multitarea = más se termina y más rápido.

Definición de Hecho (DoD)

Criterio claro de "terminado" (probado, revisado, documentado) para evitar el 90%-eterno.

Daily stand-up

15 min: qué hice, qué haré, qué me bloquea. Para sincronizar y destrabar, no para reportar al jefe.

Métricas

Burndown (trabajo restante), velocity (entregado por sprint), cycle/lead time (qué tan rápido fluye una tarea).

Gestiona bloqueos

Tu trabajo #1 como líder: remover impedimentos rápido para que el equipo avance.

4 · Manejo de personas

Delega con contexto

Delega el resultado y el porqué, no la microtarea. Da autonomía con un punto de control, no vigilancia.

Feedback con el modelo SBI

Situación → Comportamiento → Impacto: concreto y sin juzgar a la persona. Reconoce en público, corrige en privado.

1:1s regulares

Espacio quincenal por persona: cómo va, qué necesita, hacia dónde quiere crecer. Construye confianza.

Bajo desempeño

Abórdalo pronto y con datos: expectativa clara, apoyo, plan y seguimiento — no lo dejes crecer.

Motiva

Autonomía, maestría y propósito. Celebra avances y conecta el trabajo con su impacto real.

Resuelve conflictos

Escucha a ambas partes, enfócate en el problema (no en la persona) y busca el interés común.

5 · Comunicación y gobierno

RACI

Por tarea define quién es Responsable, Aprobador, Consultado e Informado. Elimina el "pensé que lo hacías tú".

Gestión de riesgos

Anticipa lo que puede salir mal (probabilidad × impacto) y ten un plan de mitigación.

Reportes ejecutivos

Al patrocinador: estado (semáforo), avance, riesgos y decisiones que necesitas — breve y claro.

Para preguntas conductuales usa STAR: Situación (contexto) → Tarea (tu responsabilidad) → Acción (qué hiciste tú) → Resultado (medible). Estructura cualquier "cuéntame de una vez que…".
Cómo lo cuentas tú: "Lidero equipos cross-funcionales de hasta 21 personas: planifico las asignaciones por capacidad y fortalezas, estimo con el equipo (story points/Planning Poker), hago seguimiento con tableros Kanban y métricas (velocity, burndown), remuevo bloqueos y desarrollo a las personas con 1:1s y feedback — entregando resultados medibles con enfoque ágil y CMMI."

Preguntas típicas de liderazgo

¿Cómo estimas un proyecto o una tarea?

Desgloso el trabajo en piezas pequeñas y estimo con el equipo por complejidad relativa (story points / Planning Poker), no en horas de una sola persona. Para lo incierto uso estimación a 3 puntos y comparo con la velocity histórica; agrego buffer para la variabilidad y re-estimo a medida que aprendemos.

¿Cómo haces seguimiento sin caer en microgestión?

Con un tablero visible, límites de trabajo en curso y una Definición de Hecho clara; el daily sincroniza y saca bloqueos. Miro métricas (burndown, cycle time) y hago 1:1s. Delego el resultado, no la microtarea: doy autonomía con puntos de control, no vigilancia.

¿Cómo manejas a alguien de bajo desempeño?

Lo abordo pronto y en privado, con datos concretos (modelo SBI). Aclaro la expectativa, entiendo la causa (¿capacidad, motivación, bloqueo?), acuerdo un plan de mejora con apoyo y hago seguimiento cercano. La clave es actuar temprano, no dejar que escale.

¿Cómo priorizas cuando "todo es urgente"?

Con criterio de valor vs. esfuerzo y MoSCoW, alineado con el objetivo del negocio y el patrocinador. Hago explícitas las compensaciones (si entra esto, sale aquello) y comunico la decisión, para que la prioridad sea transparente y no una promesa de "todo para ayer".

Preguntas típicas + tu respuesta

Respuestas cortas y correctas, ancladas en tu experiencia real. Ábrelas y practícalas en voz alta.

¿Cuál es la diferencia entre Docker y una máquina virtual?

clave Una VM virtualiza un sistema operativo completo (pesada, arranca lento). Un contenedor comparte el kernel del host y solo empaqueta la app y sus dependencias: es mucho más liviano, arranca en segundos y es reproducible. Por eso encaja perfecto con microservicios.

¿Cómo logras un despliegue sin downtime?

Con rolling updates en Kubernetes: levanta réplicas nuevas y solo cuando pasan la readiness probe empieza a enviarles tráfico, retirando las viejas gradualmente. Sumado a liveness probes para auto-reparación. Así los usuarios nunca ven una caída — justo como en la plataforma de compras y recargas que migré.

¿Cómo migraste de un monolito WebLogic a microservicios?

Con un enfoque incremental tipo strangler fig: identificar dominios, ir extrayendo servicios uno a uno sin “big bang”, comunicarlos por eventos (NATS) y REST, contenerizarlos con Docker y desplegarlos en AKS con CI/CD. Esto permitió convivir con el monolito mientras se reducía el riesgo, logrando zero-downtime y −60% en despliegues.

Explica Pod, Deployment y Service.

Pod: la unidad mínima, envuelve el/los contenedores. Deployment: declara cuántas réplicas de ese Pod quiero y gestiona actualizaciones. Service: da una dirección estable y balancea el tráfico entre esos Pods aunque cambien. El Ingress los expone al exterior por HTTP/HTTPS.

¿Cómo escalas el sistema bajo un pico de tráfico?

Con el HPA (Horizontal Pod Autoscaler), que agrega réplicas según CPU/memoria, más Redis como caché para descargar la base de datos y APIs eficientes. Esa combinación fue la que sostuvo 500 ms de respuesta bajo alto tráfico.

¿Cómo manejas configuración y secretos?

Fuera del código y de la imagen: ConfigMap para configuración y Secret para datos sensibles en Kubernetes, y Azure Key Vault para credenciales. Así la misma imagen sirve para dev, pruebas y producción cambiando solo la configuración.

¿Qué hace un Dockerfile, capa por capa?

Define cómo se construye la imagen: FROM (base), WORKDIR, COPY de dependencias, RUN para instalarlas, COPY del código y CMD para arrancar. Cada instrucción es una capa cacheada: por eso las dependencias van primero (cambian poco) y el código de último (cambia seguido), para builds rápidos.

¿Por qué event-driven y no llamadas directas entre servicios?

Porque desacopla: el emisor publica un evento y no le importa quién lo consume. Si un servicio está caído, el mensaje espera; se pueden agregar consumidores sin tocar al emisor. Da resiliencia y escalabilidad — por eso usé NATS en la arquitectura de microservicios.
